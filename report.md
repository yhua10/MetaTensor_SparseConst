# 科研课堂中期报告

**题目：面向 metatensor 的稀疏常量乘法代码生成优化研究**

**课程：科研课堂**  
**项目仓库：metatensor-main**  
**报告日期：2026 年 6 月 12 日**

## 一、研究背景与问题提出

metatensor 项目的目标是用一种较抽象的块结构描述张量、数组以及相关计算，再由后端 target 将这些结构转换为可执行代码。当前代码库中已经具备 `datadesc`、`block`、`target`、`plugin` 等核心抽象，能够用串联、并联、归约和一元运算等方式表达计算过程，并通过 C99 target 生成对应的 C 代码。这一设计使得计算表达式和最终代码之间存在可插拔的优化空间：如果某类 block 结构具有明显的数学或数据稀疏性特征，就可以在通用代码生成路径之外增加专门插件，减少不必要的计算与访存。

本阶段工作的研究问题来自一个具体但具有代表性的场景：当表达式形态为“稀疏常量数组与输入向量逐元素相乘”时，通用生成路径会按照完整数组处理所有位置，即使常量数组中大量元素为零，也仍然可能生成对零元素位置的计算与写入。对于稀疏常量系数较多的计算，这种处理方式会造成冗余乘法、冗余赋值和不必要的内存访问。由于常量数组位于 `region_direct`，其数据在代码生成时已经可见，因此理论上可以在生成 C99 代码之前识别所有非零系数，只对非零位置发射有效赋值语句，从而把运行期负担转移为编译期分析。

基于上述背景，本阶段选择“面向稀疏常量乘输入向量的 block2data 优化插件”作为研究切入点。该切入点规模适中，一方面能够覆盖 metatensor 中 block 结构识别、datadesc 维度处理、target 代码发射、插件注册和测试闭环等关键环节；另一方面也能体现项目的主要思想，即通过中间结构和插件机制把高层计算特征转化为更高效的底层代码。

## 二、研究目标与技术路线

本阶段研究目标是在 C99 代码生成路径中增加一个新的 `block2data_sparse_const` 插件，使其能够识别形如 `reduce_prod - (K & X)` 的计算结构。其中，`K` 表示编译期可见的一维常量 `datadesc`，存储区域为 `region_direct`；`X` 表示运行期输入数组，存储区域不是 `region_direct`；两者具有相同的一维索引；最终输出为新分配的 `region_main` 一维数组。插件命中后，生成代码应先为输出数组分配空间并清零，再遍历常量数组 `K` 的数据，只对非零位置生成赋值代码。对于系数为 `1.0f` 的位置，可以直接生成拷贝语句；对于其他非零系数，则生成乘法赋值语句。这样，稀疏常量中为零的位置不会在运行期产生乘法和写入。

技术路线分为四个层次。第一层是结构识别，即判断当前输入 block 是否是一个乘法归约块，并且其并联输入恰好由一个 direct 常量数据和一个非 direct 输入数据组成。第二层是合法性检查，即确认当前原型只处理一维、单 tuple 维度、32 位浮点元素，并且 `K` 与 `X` 的索引完全一致。第三层是代码生成，即通过 target 的 emit 接口生成 C99 代码，包括 `malloc`、`memset`、输入指针转换、输出指针转换以及非零系数位置的赋值语句。第四层是验证闭环，即构造最小测试程序生成 `dump_sparse.c`，再用独立验证程序比较生成代码与朴素实现的输出是否一致。

这种路线没有一开始追求完整稀疏矩阵或高维张量优化，而是先完成一个边界清晰的最小可用路径。这样做的原因是 metatensor 当前仍处于基础设施逐步完善阶段，如果直接扩展到复杂张量场景，容易把 block 识别、维度推导、内存所有权、代码生成和数值验证等问题混在一起，难以定位错误。本阶段先完成一维稀疏常量向量乘法，有助于验证插件机制是否足以承载这类优化，并为后续扩展提供可运行样例。

## 三、已完成的实现工作

本阶段新增了 `plugins/block2data_sparse_const.c`，这是本次工作的核心文件。该文件中首先实现了若干辅助函数，用于判断 block 是否为单元素 `datadesc_set`、从并联结构中抽取常量 `K` 与输入 `X`、比较一维索引是否一致、从 direct 常量数据中读取浮点值，以及根据维度 offset 计算输出数组所需字节数。主函数 `block2data_sparse_const` 负责完整的插件匹配和代码生成流程。当输入 block 不是目标结构时，函数直接返回 `0`，把处理机会交还给其他 block2data 插件；当结构匹配成功后，函数进入专门路径，生成稀疏常量乘法代码。

在代码生成过程中，插件首先通过 `target_query_type` 查询 C99 中的元素类型字符串，当前只允许 `float`，也就是 `elem_type_real` 且 `elem_size == 4`。随后插件构造一个新的输出 `datadesc`，其区域设置为 `region_main`，维度信息复制自输入 `X`，并由 `token_new()` 生成新的引用名。生成的 C99 代码会为输出数组调用 `malloc` 分配空间，并用 `memset` 将输出初始化为零。清零步骤非常关键，因为稀疏常量中为零的位置不会生成单独赋值，如果没有初始化，输出数组对应位置将保留未定义值。之后插件遍历 `K` 的所有条目，读取每个位置的常量值。若值为零则跳过；若值为 `1.0f`，生成形如 `out[i] = x[i];` 的直接拷贝；其他非零值则生成形如 `out[i] = c * x[i];` 的乘法赋值。由此，最终生成的代码只包含有效计算位置。

为了使新插件进入实际生成路径，本阶段修改了 `include/plugins.h`，加入 `block2data_sparse_const` 的函数声明；同时修改了 `targets/target_c99.c`，在 `target_c99_new()` 中注册该插件。当前注册顺序是先注册 `block2data_arithmetic`，再注册 `block2data_sparse_const`。根据项目注释“后注册的优先被匹配”，这一顺序意味着 sparse const 插件应当优先尝试匹配目标结构。如果这一调度规则在后续代码中发生变化，需要重新确认插件优先级，否则通用 arithmetic 插件可能提前接管目标表达式，使稀疏优化无法命中。

为了验证插件行为，本阶段新增了 `main_test/test.sparse_const_basic.c`、`main_test/test.sparse_const_verify.c` 和 `Makefile_test_sparse_const`。其中，基础测试程序构造长度为 16 的稀疏常量数组 `K_data`，其中 12 个位置为零，非零系数包括 `0.5f`、`1.2f`、`1.0f` 和 `-0.3f`。程序同时构造名为 `input_x` 的运行期输入 `datadesc`，再把二者组合为 `reduce_prod - (K & X)` 结构并调用 `block2data`。如果插件命中并成功生成输出 `datadesc`，测试程序会将 target 缓冲区中的 C99 代码写入 `dump_sparse.c`，并包装成 `void compute_sparse(float *input_x, float *out)` 函数。验证程序则实现了一个朴素版本 `compute_naive`，直接计算 `K_const[i] * x[i]`，并分别使用全 1 输入和递增 ramp 输入进行比较。若所有位置误差不超过 `1e-6f`，验证程序输出 `OK`。

## 四、阶段成果与技术意义

从阶段成果看，本次工作已经完成了从问题识别到原型实现的主要链路。新增插件能够在代码结构上独立于通用 block2data 路径存在，并通过 target 注册机制进入 C99 代码生成流程。新增测试也覆盖了从 metatensor 内部结构构造、代码生成、生成代码落盘、生成代码编译、结果对比的完整设想。虽然当前本机环境下尚未完成自动化测试闭环，但测试文件和验证逻辑已经具备可执行基础，只要解决编译环境兼容问题，就可以进一步确认最终运行结果。

这一成果的意义不只在于节省几个乘法，而在于验证了 metatensor 的优化思路可以落到具体实现上。稀疏常量乘法是很多科学计算和工程计算中的常见子结构，例如固定系数模板、离散算子、局部 stencil、稀疏矩阵向量乘法的简化情形等。如果常量数据在生成期可见，那么后端就不必把它视为普通运行期数组，而可以直接把常量分布转化为代码结构。本阶段实现说明，metatensor 的 `region_direct` 信息、维度描述和插件机制可以共同支持这种优化。

从代码质量角度看，本次实现采取了较保守的边界控制。插件没有试图处理所有可能情况，而是通过 `emit_assert` 明确限制支持范围。这种做法在科研课堂阶段是合理的，因为原型系统最重要的是先证明路径可行，并且在不支持的输入上尽早暴露问题。当前限制包括只支持一维数组、只支持 float、只支持单 tuple 维度、要求两个输入各自是单个 `datadesc`，并要求 `K` 与 `X` 的索引完全一致。这些限制使得代码逻辑较清晰，也便于后续逐步扩展。

## 五、测试验证情况与环境适配

本阶段规划的验证命令为 `make -f Makefile_test_sparse_const run`、`make -f Makefile_test_sparse_const verify_sparse` 和 `./verify_sparse`。预期结果是基础测试输出 `OK; dump written to dump_sparse.c`，验证程序输出 `OK`。这代表插件成功命中目标结构、生成代码成功写入 `dump_sparse.c`、生成代码能够被再次编译，并且在两组输入上与朴素实现结果一致。

实际执行时，当前 PowerShell 环境中没有名为 `make` 的命令，但已经安装了 `mingw32-make` 和 MinGW-W64 GCC 15.2.0，因此可以用 `mingw32-make -f Makefile_test_sparse_const run` 替代原命令。构建过程中暴露出若干 Windows/MinGW 环境下的兼容性问题。第一类问题是 MinGW 的 `stdlib.h` 中声明了 `itoa(int, char *, int)`，而项目 `include/target_base.h` 中原本存在同名 `itoa(int_t)`，二者发生冲突。对此，已将项目内部的辅助函数统一改名为 `mt_itoa`，避免与系统头文件冲突。第二类问题是 `src/datadesc.c` 中 `datadesc_subset_merge_search` 存在 `void **` 与 `merge_info_piece_t **` 的不兼容赋值，GCC 15 默认将其作为错误处理；当前测试 Makefile 通过加入 `-Wno-error=incompatible-pointer-types` 将其降级为警告。第三类问题是临时测试 Makefile 原本没有链接 `datacopy_foreach.o` 和 `datacopy_121_reference.o`，而 C99 target 初始化时会注册这两个插件，因此已将它们补充进 `OBJS`。

完成上述局部适配后，已在当前环境中跑通完整验证流程。`mingw32-make -f Makefile_test_sparse_const run` 能够成功生成 `dump_sparse.c` 并将其编译为对象文件，`mingw32-make -f Makefile_test_sparse_const verify_sparse` 能够成功生成验证程序，运行 `verify_sparse.exe` 后输出 `OK`。这说明当前稀疏常量代码生成路径已经在长度为 16 的测试样例上通过了结果正确性验证。

## 六、存在问题与改进方向

当前原型仍存在一些需要在后续阶段解决的问题。首先，插件匹配路径仍然较窄，只识别非常具体的 `reduce_prod - (K & X)` 结构。如果表达式经过其他优化插件重写，或者并联结构的左右顺序、包装层级发生变化，当前插件可能无法命中。后续需要将匹配逻辑从“固定形状判断”逐渐扩展为“等价结构识别”，或者在优化流程中保证目标结构在进入 block2data 前会被归一化。

其次，类型支持范围较小。目前插件只处理 32 位浮点数，常量读取、字面量输出和误差验证都围绕 float 实现。如果要支持 double 或整数类型，需要扩展 `read_K_value` 的实现方式，避免直接按 float 解释 direct 数据；同时需要根据元素类型生成不同的 C 字面量和比较逻辑。对于 double，还需要注意输出精度；对于整数类型，则需要重新定义零判断、乘法溢出和验证标准。

再次，内存所有权还需要更加统一。当前插件生成的输出数组由 `malloc` 分配，测试包装函数在复制结果后调用 `free`。这一方式适合最小验证，但在正式接入项目时，需要明确生成的 `datadesc` 生命周期由谁管理，何时释放，以及多个插件组合时临时数组如何回收。否则随着优化链变长，可能出现内存泄漏、重复释放或所有权不清晰的问题。

此外，插件中仍保留了 `fprintf(stderr, "SPARSE PATH HIT\n");` 这样的开发期调试输出。它对确认插件命中很有帮助，但正式提交或课堂展示时不应作为默认输出保留。更好的方式是将其删除，或者接入项目统一的调试开关，使正常运行输出保持干净。

## 七、后续工作计划

下一阶段将优先扩大测试覆盖范围。当前长度为 16 的测试样例已经通过验证，但仍需要增加不同稀疏率、不同非零位置分布以及边界输入的测试，并检查生成的 `dump_sparse.c`，确认零系数位置确实没有生成乘法赋值，非零系数位置生成结果符合预期。

在验证通过之后，可以继续将测试纳入已有 `main_test/Makefile` 或项目统一测试入口，避免它长期停留在独立 Makefile 中。随后可以逐步扩展插件能力：第一步支持更多非零系数模式和更严格的边界测试；第二步支持 double 类型；第三步考虑二维或多维场景；第四步把该优化与 `optimize_common_factor`、`optimize_map_reduce` 等已有优化方向结合起来，使 metatensor 对更复杂的科学计算表达式也能产生更紧凑的 C99 代码。

从科研训练角度看，后续还应补充更系统的性能评价。当前工作主要验证“能否生成正确代码”，但还没有量化“优化后节省多少运行期操作”。下一阶段可以设计不同稀疏率的常量数组，分别统计生成代码行数、乘法次数、写入次数和运行时间。通过与通用生成路径或朴素实现对比，可以更清楚地说明该插件在什么稀疏率下收益明显，以及在什么情况下生成代码体积增加可能抵消优化收益。

## 八、阶段总结

本阶段围绕 metatensor 的 C99 后端代码生成路径，完成了一个面向稀疏常量乘输入向量场景的优化原型。工作内容包括分析 metatensor 当前 block2data 可处理结构，明确 `region_direct` 常量数据在生成期可见这一优化依据，新增 `block2data_sparse_const` 插件，在 C99 target 中注册插件，并补充基础测试与结果验证程序。该工作已经形成了较完整的实现链路，能够作为后续稀疏计算优化、常量传播优化和代码生成特化优化的基础。

当前最大的不足是测试覆盖仍然较小，主要验证了一维 float 稀疏常量向量乘法这一最小场景。尽管如此，新增插件的结构、注册、代码生成和验证闭环已经基本完成，后续可以继续开展更多输入规模和稀疏率下的正确性验证与性能评估。总体而言，本阶段工作达到了中期目标：没有停留在概念分析，而是围绕一个真实代码库完成了可追踪、可扩展、可继续验证的研究原型。
