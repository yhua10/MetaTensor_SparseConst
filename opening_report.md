# 科研课堂开题报告

**题目：面向 MetaTensor 框架的稀疏常量乘法代码生成优化研究**

**课程：科研课堂**  
**项目仓库：metatensor-main**  
**开题日期：2026 年 6 月 12 日**

## 一、项目背景

科学计算、数值模拟和工程计算中经常出现大量张量、数组和矩阵相关操作。传统做法通常是由程序员直接使用 C、C++、Fortran 或 Python 数值库手写计算过程，但当计算表达式变得复杂时，手写代码往往同时面临两个问题：一是表达式本身和底层循环、数组访问、临时变量管理混在一起，可读性和可维护性下降；二是程序员需要手工发现并利用数据结构中的特殊性质，例如常量、稀疏性、重复计算和维度对应关系，否则生成的代码容易包含冗余运算。

MetaTensor 项目的基本思路是将张量计算从直接手写底层代码中抽象出来，先用一种中间结构描述数据和计算，再由后端 target 生成具体目标语言代码。这样做的优势在于，计算表达式一旦被转化为统一的中间表示，就可以在代码生成前进行结构分析和优化。例如，如果某个数组是编译期可见的常量数组，并且其中大量元素为零，那么后端就不必生成完整的逐元素乘法代码，而可以只对非零位置生成有效计算。这样的优化如果完全依赖人工手写，既容易出错，也难以在不同计算表达式之间复用；如果放入框架插件中，则可以成为稳定的自动化优化能力。

本课题以当前 metatensor-main 代码仓库为研究对象，选择“稀疏常量乘输入向量”作为具体切入点，研究如何在 MetaTensor 的 block2data 代码生成路径中识别这一类特殊计算结构，并为 C99 后端生成更紧凑的代码。该课题规模适中，但覆盖了框架理解、数据描述、计算图结构识别、插件注册、C99 代码发射和测试验证等完整环节，适合作为科研课堂中的实践型研究题目。

## 二、MetaTensor 框架简介

从当前项目代码看，MetaTensor 框架主要由 `datadesc`、`block`、`target` 和 `plugin` 几类核心抽象组成。`datadesc` 用来描述数据本身，包括数据所在区域、元素类型、元素大小、维度信息、索引和 offset 等。它并不只是一个普通数组指针，而是携带了数组在计算表达式中的结构信息。通过 `datadesc_set`，框架可以把一个或多个数据描述组合起来，作为计算块的输入或输出。

`block` 用来描述计算结构。根据项目 memo 中的约定，当前框架中的基本计算元素包括数据块 `[D]`、归约运算 `[R]`、一元运算 `[U]`、导线 `[n-m]` 和生成器 `[G]` 等。不同 block 可以通过串联 `-` 和并联 `&` 组合起来。串联表示一个计算结果继续进入下一个计算，求同并联表示多个计算分支在维度和索引意义上并行组合。通过这些基本连接方式，框架可以表达诸如一元运算映射、归约求和、归约乘法、数据重命名和生成循环范围等结构。

`target` 表示代码生成后端。当前项目中已有 C99 target，它负责查询目标语言中的数据类型、管理输出代码缓冲区、生成变量名，并将上层 block 结构转化为 C 代码。`plugin` 则是框架实现可扩展优化的重要机制。项目中已经存在若干 datacopy 插件、block2data 插件和 optimize 插件，不同插件负责识别不同的结构并尝试生成更合适的代码。插件采用链式注册方式，当一个插件不能处理当前输入时返回失败，框架继续尝试下一个插件；当某个插件成功匹配时，就由它完成相应代码生成。

本课题重点关注的是 block2data 路径。block2data 的作用可以理解为：给定一个表现为数据输出的计算 block，把它转换成一个新的 `datadesc_set`，并在 target 中发射相应的底层代码。根据 memo 中的设计，复杂 block 最终可以被转化为带有显式读操作、写操作和循环生成器的标准形式，再通过拓扑排序或递归分治方式输出代码。本课题不直接完成通用 block2data 的全部设计，而是在其插件机制上增加一个面向稀疏常量乘法的专门优化路径。

## 三、问题分析

本课题关注的具体表达式可以记为 `reduce_prod - (K & X)`。其中，`K` 是编译期可见的常量数据，存储区域为 `region_direct`；`X` 是运行期输入数据，存储区域不是 `region_direct`；二者具有相同的一维索引；归约乘法 `reduce_prod` 表示在相同索引位置上进行乘法组合。从数学意义上看，这一表达式对应逐元素计算 `Y[i] = K[i] * X[i]`。

如果使用通用代码生成方式，后端通常会为每一个位置生成乘法和写入逻辑。对于普通稠密数组，这样做是合理的；但当 `K` 是稀疏常量数组时，许多位置满足 `K[i] = 0`，这些位置的输出必然为零。既然 `K` 位于 `region_direct`，它的实际数值在代码生成阶段已经可见，那么继续为零系数位置生成运行期乘法就没有必要。更合适的方式是：在生成代码时提前扫描常量数组，先把输出数组初始化为零，然后只对 `K[i] != 0` 的位置生成赋值语句。若 `K[i] = 1`，甚至可以省略乘法，直接复制输入值。

这个问题的难点并不在数学公式本身，而在如何把公式和框架内部结构对应起来。插件需要确认当前 block 的外形确实是乘法归约结构，需要从并联输入中区分哪个是常量 `K`、哪个是输入 `X`，需要确认两者的维度和索引能够一一对应，还需要根据 target 的代码发射接口生成合法 C99 代码。若匹配过于宽松，可能会错误优化不等价表达式；若匹配过于狭窄，则只能处理极少数手写测试样例，难以体现框架优化价值。因此，本课题采用“先明确边界、再逐步扩展”的策略，先完成一维 float 场景下的正确原型。

此外，当前项目还存在一定工程环境问题。项目代码中有些写法在较早 C 编译器中可能只是警告，但在 Windows MinGW-W64 GCC 15.2.0 环境下会表现为编译错误，例如项目内部 `itoa` 与 MinGW 系统头文件中的 `itoa` 同名冲突，以及部分指针类型不兼容问题。课题实施过程中需要同时关注算法优化和工程可运行性，确保研究成果不是停留在设计文档中，而是能在本地实际编译和验证。

## 四、课题目标

本课题的总体目标是在 MetaTensor 的 C99 代码生成路径中实现一个面向稀疏常量乘法的 block2data 优化插件，验证框架能够利用 `region_direct` 常量数据中的稀疏性，在生成期减少运行期冗余计算。

具体目标包括以下几个方面。首先，需要理解并梳理 MetaTensor 当前的核心数据结构和 block 组合方式，明确 `datadesc`、`block`、`target` 和 `plugin` 在代码生成流程中的职责。其次，需要设计目标表达式的结构匹配条件，能够识别 `reduce_prod - (K & X)` 这一类一维稀疏常量乘输入向量结构，并正确区分 direct 常量数据和运行期输入数据。再次，需要在 C99 target 中完成代码生成，使生成代码只对非零常量位置执行赋值，对零系数位置不生成乘法计算。最后，需要补充测试程序和验证流程，通过朴素实现对比生成代码的输出结果，证明优化后的代码在目标场景下保持计算正确性。

在支持范围上，本课题初期限定为一维数组、32 位浮点元素、单 tuple 维度、一个 direct 常量输入和一个非 direct 运行期输入。这样的限定不是最终目标，而是为了先建立可验证的最小闭环。在最小闭环完成后，再考虑扩展到更多数据类型、更多维度和更复杂的表达式结构。

## 五、研究方案与技术路线

本课题的研究方案采用“框架分析、插件设计、代码实现、环境适配、测试验证、总结扩展”的路线展开。

第一阶段是框架分析。通过阅读 `include/`、`src/`、`plugins/` 和 `targets/` 目录中的代码，理解 MetaTensor 如何用 `datadesc` 表示数据，用 `block` 表示计算，用 `target_c99` 发射 C99 代码，并了解已有插件的注册和调用方式。同时结合项目 memo 中关于基本程序、block2data 标准形式和拓扑代码生成的说明，确定本课题适合以 block2data 插件作为入口，而不是直接修改通用 target 或重写整体代码生成逻辑。

第二阶段是插件匹配逻辑设计。目标插件命名为 `block2data_sparse_const`。它首先判断输入 block 是否为串联结构，并且串联左侧是否为乘法归约 `block_builtin_reduce_prod`；随后检查右侧是否为求同并联结构，并从两个并联分支中抽取 `K` 和 `X`。其中一个分支必须是单元素 `datadesc_set` 且 region 为 `region_direct`，另一个分支必须是单元素 `datadesc_set` 且 region 不是 `region_direct`。在确认结构匹配后，还要检查元素类型、维度数量、tuple 数量和索引一致性，防止对不满足条件的输入进行错误优化。

第三阶段是 C99 代码生成设计。插件匹配成功后，新建一个输出 `datadesc`，其区域为 `region_main`，维度描述复制自输入 `X`。代码生成时先调用 `malloc` 为输出数组分配空间，再调用 `memset` 将输出初始化为零。随后在生成期遍历常量 `K` 的所有条目，读取每个位置的浮点系数。如果系数为零，则不生成任何赋值语句；如果系数为 `1.0f`，生成直接拷贝语句；如果系数为其他非零值，生成乘法赋值语句。这样，生成代码的运行期操作数量与常量数组中非零元素数量相关，而不再与数组总长度完全绑定。

第四阶段是测试与验证。构造一个长度为 16 的一维 float 测试样例，其中常量数组包含多个零元素和少量非零元素。基础测试程序负责构造 `datadesc` 和 block，调用 block2data，并将生成的代码写入 `dump_sparse.c`。验证程序提供朴素计算函数 `compute_naive`，分别使用全 1 输入和递增输入调用朴素函数与生成函数，并比较两者输出是否一致。若误差在设定阈值内，则说明生成代码在该样例下正确。

第五阶段是环境适配与结果分析。由于当前 Windows 环境使用 MinGW-W64 GCC 15.2.0，课题需要处理 `make` 命令差异、项目内部函数命名冲突和旧代码警告升级等问题。适配原则是不大幅改变用户电脑环境，不进行全局安装和系统级修改，而是在项目内部 Makefile 和源码命名上做局部调整，使测试流程能够在当前仓库内稳定运行。验证通过后，再根据生成代码和测试结果总结优化效果、支持边界和后续扩展方向。

## 六、预期成果

本课题预期形成一套可以直接运行和展示的研究成果。首先，在代码层面完成 `block2data_sparse_const` 插件，实现对一维 float 稀疏常量乘输入向量结构的识别和 C99 代码生成。其次，在框架集成层面完成插件声明与 target 注册，使该插件能够通过现有 block2data 调度机制被调用，而不是作为孤立实验代码存在。再次，在测试层面提供基础生成测试、结果验证测试和专门 Makefile，使课题成果可以通过命令行复现。

在文档层面，预期形成开题报告、中期报告和后续总结材料，清楚说明项目背景、框架结构、问题来源、技术路线、实现方法、验证结果和后续改进方向。对于课堂汇报，预期能够展示一个完整故事：MetaTensor 用中间结构表达计算；稀疏常量乘法中存在可利用的生成期信息；通过插件识别这一结构并生成特化 C99 代码；最终用测试证明生成结果与朴素计算一致。

在研究价值层面，本课题预期证明 MetaTensor 框架的插件机制可以承载面向数据特征的代码生成优化。虽然本课题只选择了一维稀疏常量乘法作为原型，但同样的思路可以扩展到固定系数模板、稀疏矩阵向量乘法、离散算子和更多科学计算表达式中。后续如果继续推进，可以进一步支持 double 类型、多维数组、更多归约结构以及基于代价模型的自动选择策略，使 MetaTensor 在科学计算代码生成方面具备更强的实用性。

## 七、可行性分析

本课题具有较好的可行性。首先，项目代码中已经存在 target、block2data 和 plugin 的基础设施，不需要从零搭建编译器或代码生成框架。其次，目标场景边界清晰，输入结构可以明确限定为 `reduce_prod - (K & X)`，数据类型和维度也可以先限定在一维 float 范围内。再次，验证方式直接可靠，只需要将生成代码与朴素逐元素乘法结果比较即可判断正确性。

主要风险在于框架内部结构仍处于探索阶段，部分代码和 memo 中的设计尚未完全统一，通用 block2data 路径也还没有覆盖所有复杂表达式。因此本课题不会一开始追求通用化，而是以专门插件方式实现最小闭环。另一个风险是 Windows 下编译环境与项目原有代码存在兼容问题，但该风险可以通过仓库局部修改和测试 Makefile 适配解决，不需要对系统环境做大规模调整。

## 八、进度安排

前期主要完成项目结构阅读和问题定义，重点理解 `datadesc`、`block`、`target_c99` 与现有插件机制，并确定稀疏常量乘法作为研究切入点。中期完成插件原型实现，包括结构识别、合法性检查、输出 `datadesc` 构造、C99 代码发射和插件注册。随后完成测试程序和验证程序，解决本地 MinGW/GCC 环境下的构建问题，确保测试能够跑通。后期主要进行结果总结、报告完善和汇报材料准备，并根据时间补充更多测试样例或性能统计。

## 九、总结

本课题围绕 MetaTensor 框架中的 C99 代码生成优化展开，选择稀疏常量乘输入向量这一具体场景作为研究对象。该场景能够体现框架中间表示的价值：当常量数据在生成期可见时，后端可以利用其稀疏性生成更简洁的运行期代码。本课题计划通过新增 block2data 插件完成结构识别和特化代码生成，并通过独立测试验证结果正确性。预期成果既包括可运行代码，也包括可用于课堂汇报的研究报告和技术说明，为后续更复杂的 MetaTensor 优化研究打下基础。
